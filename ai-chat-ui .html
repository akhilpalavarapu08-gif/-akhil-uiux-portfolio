Overview

Why ARIA?

AI tools were everywhere — but their interfaces weren't designed for the people actually using them. Cluttered layouts, poor keyboard navigation, low contrast ratios, and zero attention to motion made them frustrating and exhausting, especially for users with accessibility needs.

ARIA was my answer: a clean, focused chat interface that puts the conversation first and every design decision in service of the user — not the technology.

The Problem

Existing AI chat interfaces prioritised feature density over usability. Users with visual or motor impairments were effectively excluded, and the cognitive load of most UIs made sustained use tiring.

The Solution

A minimal, WCAG 2.1 AA-compliant chat interface with a considered motion language, keyboard-first navigation, and a component system built to scale across web and mobile.

Process

How it was designed

An 8-week sprint from research to handoff — every phase focused on validating assumptions before moving forward.

01

Discover

Audited 8 existing AI chat products. Ran 12 user interviews. Identified 3 core pain-points: cognitive overload, poor accessibility, and inconsistent feedback.

02

Define

Synthesised research into 2 primary personas. Mapped the core conversational flow. Set accessibility targets: WCAG 2.1 AA minimum, keyboard-navigable throughout.

03

Design

Low-fi wireframes → mid-fi prototype → high-fi Figma component library. 3 rounds of design critique with iteration between each.

04

Test & Ship

Unmoderated usability tests with 8 participants. Accessibility audit with screen reader. Iterated on findings and handed off to development.

Design Decisions

Key choices & why

Typing indicators with purposeful delay

Rather than instant responses, ARIA shows a 400–800ms typing animation before replying. This mirrors natural conversation pacing, reduces perceived abruptness, and gives users a moment to read their own message — tested to reduce re-read rates by 22%.

Conversation threading in a sidebar

Power users need to manage multiple sessions without losing context. A collapsible sidebar groups threads by recency and allows renaming — keeping the main chat surface completely distraction-free for focused work.

Colour contrast above 4.5:1 everywhere

Every text element, icon, and interactive component meets WCAG 2.1 AA. The deep navy + violet palette was chosen specifically because it achieves high contrast while remaining visually premium — proving accessibility and aesthetics aren't mutually exclusive.

Minimal input surface, maximum focus

The message input is a single line that expands on multi-line content. No toolbar clutter. Keyboard shortcuts handle formatting. This keeps first-time and power-user flows identical — no learning curve.

Reflection

What I learned

Accessibility raises the bar for everyone. Designing for users with visual impairments led to better contrast, clearer hierarchy, and faster comprehension for all users — not just those who needed it.

Motion is a communication tool, not decoration. Every animation in ARIA carries meaning — the typing indicator communicates processing, the slide-in communicates arrival. Purposeless animation was cut in round 2.

Test earlier than feels comfortable. I ran the first usability session with mid-fi wireframes — and caught a fundamental navigation issue that would have been expensive to fix at high-fi stage.

A design system is worth the upfront cost. Building a component library from day one made the 3 rounds of iteration dramatically faster — changing the primary colour took 2 minutes, not 2 hours.
